# 流程模型

> 官方文档：[流程模型 | Electron](https://www.electronjs.org/zh/docs/latest/tutorial/process-model)

Electron 继承了来自 Chromium 的多进程架构，这使得此框架在架构上非常相似于一个现代的网页浏览器。

## 为什么不是一个单一的进程？

早期浏览器通常使用单个进程处理所有功能。虽然开销较少，但一个网站的崩溃会影响到整个浏览器。Chrome 团队决定让每个标签页在自己的进程中渲染，从而限制错误的影响范围。

## 多进程模型

Electron 应用程序的结构非常相似。作为应用开发者，你将控制两种类型的进程：**主进程**和**渲染器进程**。

## 主进程

每个 Electron 应用都有一个单一的主进程，作为应用程序的入口点。主进程在 Node.js 环境中运行，具有 `require` 模块和使用所有 Node.js API 的能力。

### 窗口管理

主进程使用 `BrowserWindow` 模块创建和管理应用程序窗口。每个 `BrowserWindow` 实例在单独的渲染器进程中加载一个网页。

```js
const { BrowserWindow } = require('electron')
const win = new BrowserWindow({ width: 800, height: 1500 })
win.loadURL('https://github.com')
const contents = win.webContents
```

### 应用程序生命周期

主进程通过 `app` 模块控制应用程序的生命周期：

```js
app.on('window-all-closed', () => {
  if (process.platform !== 'darwin') app.quit()
})
```

### 原生 API

主进程添加了自定义 API 来与用户的操作系统进行交互，例如菜单、对话框以及托盘图标。

## 渲染器进程

每个 `BrowserWindow` 会生成一个单独的渲染器进程，负责渲染网页内容。渲染器进程遵循 Web 标准：

- 以 HTML 文件作为入口点
- 使用 CSS 对 UI 添加样式
- 通过 `<script>` 元素添加 JavaScript 代码

渲染器无权直接访问 `require` 或其他 Node.js API。要在渲染器中包含 NPM 模块，需要使用 webpack 或 parcel 等打包工具。

## Preload 脚本

预加载脚本包含了那些执行于渲染器进程中，且先于网页内容开始加载的代码。这些脚本虽运行于渲染器的环境中，却因能访问 Node.js API 而拥有了更多的权限。

```js
const win = new BrowserWindow({
  webPreferences: {
    preload: 'path/to/preload.js'
  }
})
```

由于默认启用了 `contextIsolation`，不能直接在预加载脚本中将变量附加到 `window` 上，应使用 `contextBridge` 模块：

```js
const { contextBridge } = require('electron')
contextBridge.exposeInMainWorld('myAPI', { desktop: true })
```

## 实用进程

每个 Electron 应用程序都可以通过 `UtilityProcess` API 生成多个子进程，用于托管不受信任的服务、CPU 密集型任务等。

## 进程相关模块别名（TypeScript）

- `electron/main`：主进程模块的类型
- `electron/renderer`：渲染进程模块的类型
- `electron/common`：主进程和渲染进程内都能运行的模块的类型

```js
const { shell } = require('electron/common')
const { app } = require('electron/main')
```
